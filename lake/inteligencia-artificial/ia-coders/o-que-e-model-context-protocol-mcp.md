O que é MCP? Vamos lá. MCP é o USB dos agentes, o padrão aberto para conectar IA ao mundo. Para você entender o que está acontecendo, para que esse cara nasceu, antes de mais nada, eu quero te lembrar como era a era pré-MCP. Se você quisesse conectar com uma API, por exemplo, ou várias APIs para o seu agente consumir, você teria que criar uma interface para cada uma dessas APIs, por exemplo, para o Gmail, para o Drive, para o Slack, para o Notion, e manter isso dentro do seu repositório. Eu fiz isso, inclusive, lá atrás, antes de MCP existir, e era um tênis complicado. Você tinha que definir do lado de cá as tool callings, ou a maneira como o LLM iria identificar cada uma dessas chamadas, e para isso você tinha que fazer uma verdadeira engenharia para o LLM conseguir entender e fazer a chamada das tools corretas. E o conceito de tool calling nem era tão forte ainda naquela época. Ou seja, talvez você teria 16 integrações para 4 apps versus 4 serviços diferentes, e aqui cada app usando uma inteligência artificial, um client ali, diferente. Para resolver essa maçaroca toda, aí a Anthropic lançou o conceito de MCP. Ou seja, você constrói um MCP, esse MCP vai ser o protocolo que vai facilitar a conexão de um server com o client. Então, eu posso ter um MCP server que vai para o lugar na API do Gmail, para o Gmail desenvolver esse MCP e fornecer para o mercado, e a gente simplesmente se conecta a ele, o MCP do Google Drive. Eu simplesmente conecto e uso esse cara e por aí vai. Ou seja, a ideia é que assim como o HTTP, qualquer navegador de internet consegue resolver o HTTP, o MCP também, qualquer AI agent consiga resolver o MCP, independente de quem está fornecendo ele. Isso é super importante, principalmente quando a gente precisa de lidar com informações em bancos de dados e autenticação. A gente vai entender aqui já já. Então, definição por padrão, MCP é um protocolo aberto que foi criado pela Anthropic em novembro de 2024, que padroniza como aplicações de IA descobrem e conectam ferramentas, dados e fluxos externos. Ele é baseado em JSON e RPC 2.0. Client serve como transportes padronizados em STDio, quando você está usando o MCP local, ou HTTP ou Service Center Events. Qualquer cliente fala com qualquer server. A arquitetura básica funciona da seguinte maneira. Você tem o LLM, o Harness, um Code Agent, que se conecta com o Client, que foi instalado naquele Code Agent. Você pode ter uma relação 1 para N, então cada Harness pode ter N MCP Clients conectados. E esse é o Client. Esse MCP Client, via JSON ou RPC, vai se conectar com o MCP Server. A ideia do MCP Server é do Gmail, do Notion, do Contextify, que seriam algumas propostas aqui para você. Quais são as três primitivas do MCP Server? Vamos lá, são três muito simples de aprender. A primeira são as Tools, a segunda Resources e a terceira Prompts. Tools, o modelo controla a chamada dessas Tools. Então, são funções que o LLM pode chamar no seu esquema de Input e Output e quais são os efeitos que elas produzem. Por exemplo, você tem a Tool Send E-mail. A Tool Create Issue. A Tool Run Carry. Recursos são dados que o Host escolhe anexar ao contexto. Então, eu vou fornecer alguns recursos em arquivos Markdown, instruções, que vão ficar disponíveis através de URIs, que são somente leitura, que não tem nenhum efeito colateral. Eu posso ter, por exemplo, Readme.md, um DB User's Schema, um Git Branch Main e por aí vai. E os Prompts são controlados pelo usuário, são templates reutilizáveis que o usuário invoca.
