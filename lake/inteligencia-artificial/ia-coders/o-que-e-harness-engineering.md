# Transcricao — O que e Harness Engineering

> Conteudo BRUTO (lake). Insumo para curadoria pela skill prof-toni.

Bora falar de Harness Engineering. Então, qual que é o conceito? Harness é toda a infraestrutura ao redor do LLM. As ferramentas que ele chama, o ambiente onde ele executa, a memória, o loop de decisão, os guardrails, o feedback. Ou seja, o modelo sozinho, ele é só o cérebro. Eu gosto de fazer analogia com o carro. Então imagina que o modelo sozinho, ele é o motor do carro. Assim como você tem ali dentro da Ferrari um baita do motor, você, na teoria, conseguiria tirar o motor da Ferrari e botar no Fusca. Vai ficar bom? Não necessariamente. Talvez o Harness não aguenta. O Fusquinha vai peidar. Mas a mesma ideia de você instalar o motor de Opala dentro de um outro carro, motor de Gol dentro de Fusca e por aí vai. A ideia é que o Harness é a carroceria que garante que você vai ter mais conforto durante a sua viagem. Ou seja, se você está desenvolvendo software com Cloud Code, Cloud Code é um Harness bem confortável. Você tem sub-agents, agent teams, um sistema de memória bonitinho. Agora, se você está usando um negócio mais raiz, o P-agent, cruzão, o P, você está usando um negócio mais bruto. Para você ter mais conforto, tem que instalar algumas extensões. Entendeu a ideia? Então o Harness é justamente isso que envolve o LLM. Por que ele precisa existir? Porque se você está só com o LLM fazendo uma troca de ideia, você volta à era do chat EPT para codificação. Você mandava ali ao chat EPT, faz para mim essa classe Java, esse componente React, a gente devolvia, você copiava e colava dentro do editor de código. Então o Harness vem justamente... para você não precisar de fazer isso na mão, para ele trazer esse desenvolvimento mais agêntico. E aí a gente começa a observar exatamente isso. O que é o desenvolvimento agêntico? Você tem o Agent Loop, que é o coração do Harness, que basicamente ele vai observar todo o input que está acontecendo, então o estado atual, os resultados das tools, tudo que está acontecendo. Vai pensar, vai ter o reasoning, vai decidir qual que é a próxima ação. Se é fazer uma chamada de tool, então ele vai agir. Se é executar o código, se é fazer um read, qualquer coisa nesse sentido, e vai trazer o resultado. Se o resultado satisfaz a condição, então ele está pronto. Se não faz, ele volta e cai no loop de novo. Então o feedback no final, ele vira a próxima observação e por aí ele vai entendendo se ele precisa continuar nesse loop ou não. Beleza? Então isso é o que acontece dentro de um Harness. Isso é o que acontece hoje dentro do Cloud Code, por exemplo. O Harness, via de regra, a diferença dele para o modelo, é que o Harness chama o modelo em loop, gerencia tools e MCP servers, mantém estado e memória, valida a prova e interrompe. Ele é stateful, ele é quem dá ao modelo a capacidade de agir no mundo. Agora, o modelo, aí a gente está falando, por exemplo, de Opus 4.7, GPT 5.5 e por aí vai, ele recebe o contexto, produz tokens, não tem memória persistente, não executa nada sozinho e não vê o mundo real. Ele é stateless, mas determinístico no comportamento diário. É um modelo de I.O. puro, beleza? Então essa é a diferença principal entre os dois. Anatomia de um Harness em sete componentes. Primeiro, tool e functions. O que o agente pode fazer, por exemplo, editar arquivos, bash, websearch, MCP servers. Cada tool tem um esquema, descrição e o que ele vai executar. Outro ponto, memória e estado, memory state. O que o agente lembra no short term, uma conversação, por exemplo, dentro da janela de contato, no long term pode ser um vector DB, scratchpad e summarization. No caso de gerenciamento de contexto, a gente também está falando que é uma parte importante da Anatomia de um Harness. Então o que o modelo vê agora? O system prompt, skills, as descrições das tools, o histórico relevante para aquela ação que ele está executando, compactação, write just in time e por aí vai. É justamente o gerenciamento de contexto. Outra parte importante da Anatomia de um Harness, sandboxing. Enviroment, aonde que o agente está sendo executado. Se é um container, VM, se é no profile system, network, os limites de recurso, isolamento igual a segurança. Ou seja, se você consegue isolar o seu code agent, você vai ter um pouco mais de segurança com ele. É o que a gente fala, por exemplo, da própria ideia também do Harness open crawl. Não instala o open crawl no seu computador pessoal, isola ele em um ambiente mais seguro. Guard rails é uma parte importante da Anatomia de um Harness e também as políticas. O que o agente não pode fazer? Ele pode abrir um .env? Não pode. Ele pode comitar e fazer push por produção sozinho? Não pode. Então, permissões, rate limits, cost caps, necessidade de aprovação de humano, ou seja, human approval gates, lista de permissão, o que ele pode e não pode fazer? Entra dentro dos guard rails. Título de comparação do Cloud Code, você tem ali esses guard rails bem claros. E quando você quer fazer no modo doidão, você bota no dangerous skip permissions ou YOLO. Essa ideia dentro dos coding engines para você desativar esses guard rails ou a maioria deles. Na observabilidade, é basicamente para você observar o que já rolou. Traces, logs, uso de tokens, uso de two calls, evals, replay. Sem isso, basicamente, você está voando às cegas. Então, isso aqui vai te trazer clareza. A maioria das ferramentas trazem isso para você hoje também. Outra parte importante da Anatomia de um Harness são os sensores. Ele é o que ativa os outros workflows e scripts. Então, basicamente, sempre que acontece alguma coisa, ele que vai ativar. Opa, preciso chamar aqui um revisor, um agente revisor, um subagent e por aí vai. Então, são os sensores. Isso aqui compõe a Anatomia de um Harness em sete componentes. E a topologia de um Harness? Vamos lá, rapidamente. A topologia consiste no single agent loop, no orchestrator mais subagent, no pipeline DAD e também no autonomous agent. Vamos lá. Primeiro ponto, single agent loop. Um agente, muitas chamadas. Simples de operar. Beleza? Por exemplo, cursor Cloud Code, bom para codificação. Orquestrador mais subagent. Aqui a ideia é você ter um agente orquestrando e outros subagentes trabalhando. Você faz isso no Cloud Code também. Agora você já tem a evolução das ferramentas de codificação nesse sentido. Mas tem a ideia do Research, do Deep Agents aqui também para você trabalhar. Ou seja, você vai ter um agente pai orquestrando outros agentes filhos para trabalharem para ele. E aí ele devolvendo a função para esse agente pai. Pipeline DAD são passos fixos. Cada um com um prompt especializado. Previsível, auditável. Ele é bom para workflows, por exemplo, com spec. Os autonomous agents são os que decidem tudo. Roda muitas steps e com pouca supervisão. Você tem um alto risco, mas também tem uma recompensa alta. Porque o risco alto é dele vacilar no meio do caminho, dele alucinar e aí você ficaria sem referência de quando ele alucinou e poder parar ele. Mas o alto reward é que se detrair, tudo certo, você economizou tempo de execução. Aí tem alguns exemplos. Tenha sempre consciência de que sempre que você puder estar envolvido, o custo de retrabalho é menor, beleza? Falhas comuns em Harness ruim. Você vai sempre observar isso aqui. Tem até uma palavrinha solta, eu explico pra ela que se fizer sentido. Mas basicamente, loop infinito é uma falha muito comum. Sem ter um Stop Condition no Agent Loop, você vai ficar gerando custo da unidade de tokens. Context Poisoning, que é a possibilidade de erros antigos que ficam dentro do histórico e confundem o LLM. Então, às vezes você esqueceu de dar um reset, um compact, na janela de contexto do LLM. Ele manteve ali uma coisa que não tem nenhuma relação com o que você está desenvolvendo. Isso pode ser um problema no Harness. E se o Harness é ruim, ou seja, se ele não trata isso por conta própria, ele também pode trazer isso. Tool Overload. Mais de 50 tools no Promot, por exemplo. O modelo pode escolher a tool errada. Isso é um problema comum de Harness ruim. Provavelmente não é o Harness que você está utilizando. Mas se você estiver usando a ferramenta e ela estiver mandando mal, pode ser isso. Silent Failure é quando tudo falha. Desculpa, a tool falha e o modelo inventa o resultado e segue em frente. Fiz a chamada aqui no Atul de websearch, buscou bosta nenhuma e falou com você que buscou e seguiu em frente. Alguns desses erros eram muito comuns lá atrás no Cloud Code. Hoje nem tanto. Mas quando você tiver de optar por uma ferramenta, um Harness, olhe para esses quatro caras. Se algum deles estiver acontecendo, é um sinal de alerta vermelho, porque isso significa que o Harness provavelmente é ruim. É que aquela palavrinha solta, que é o Context Rot, é o problema que está muito relacionado à ideia do Context Poisoning. Então, quanto mais você usa uma janela de contexto, maior a probabilidade de um LLM alucinar. Isso a gente chama de Context Rot. Então, quando você compacta o contexto ou cria um novo chat, por exemplo, você reduz o problema do Context Rot, que é justamente essa probabilidade de quanto mais você usa o LLM, mais a janela de contexto fica sobrecarregada, mais burro o LLM fica.
