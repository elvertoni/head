Seção 01

O que vamos construir

Quando você manda mensagem no WhatsApp Web, ela voa pro outro lado do mundo sem a página recarregar uma única vez. O Spotify troca de música, o Twitter atualiza o feed, o iFood recalcula o frete — tudo sem o navegador respirar. Quando acabar esta aula, você nunca mais vai olhar pra um formulário web do mesmo jeito.

Hoje começa um projeto que vai te acompanhar pelas próximas oito aulas — e que, se você quiser, vira seu primeiro item de portfólio sério: o Caderno de Estudos. Um app pessoal onde você registra suas sessões de estudo (matéria, tempo, anotação), acompanha sua evolução e, até o final da trilha, estará publicado online com login, gráficos e dados privados por usuário. Hospedado, com URL pública, pronto pra postar no LinkedIn.

No primeiro dia, você constrói a peça mais fundamental: o formulário inteligente que recebe a sessão e a exibe em tela como mágica, sem nunca recarregar a página.

✓

Resultado final

Um formulário que captura matéria, tempo em minutos e anotação. Ao clicar em Salvar sessão, a página NÃO recarrega. Um card verde aparece animado acima do formulário com os dados preenchidos, e o formulário se limpa sozinho, pronto para a próxima sessão. Quanto mais sessões você adiciona, mais cards empilham — o mais novo no topo.

Seção 02

A analogia do garçom

Imagina que você está num restaurante e pede um suco de laranja. O garçom anota e, antes que você consiga dizer "espera, sem açúcar", ele já saiu correndo pra cozinha. Resultado: chega um suco doce que você não queria, e agora é tarde.

Formulários HTML funcionam exatamente assim por padrão. O usuário clica em Enviar e o navegador IMEDIATAMENTE empacota os dados, dispara uma requisição HTTP para o endereço do atributo action e recarrega a página inteira. Tudo que você queria fazer antes — validar, transformar, mostrar em tela, salvar em algum lugar — virou pó.

A função preventDefault() é você segurando o braço do garçom: "espera aí, deixa eu falar com você primeiro". O navegador para. O JavaScript assume o controle. E a partir dali, você decide o que fazer com aqueles dados.

i

O ciclo padrão de um formulário HTML

Sem JavaScript interceptando, o navegador faz quatro coisas em sequência quando você clica em Enviar: empacota os campos do formulário, dispara uma requisição HTTP para o endereço do atributo action, espera a resposta do servidor e recarrega a página inteira.
Tudo que estava na memória do JavaScript se perde no caminho. O preventDefault interrompe esse ciclo logo no início. A partir daí, o JS assume e você manda em tudo.

3. As três linhas que mudam tudo

Agora a parte que importa. Crie o app.js:

javascript 
const form = document.getElementById('form-sessao');

form.addEventListener('submit', function(evento) {
  evento.preventDefault();

  const dados = new FormData(form);
  const sessao = Object.fromEntries(dados);

  console.log(sessao);
});

Abra o index.html com Live Server, abra o DevTools (F12) na aba Console, preencha o formulário e clique em Salvar sessão. Você vai ver no console:

txt 
{ materia: "Matemática", tempo: "45", anotacao: "Função quadrática" }

Olha o que aconteceu. Você capturou os dados de um formulário inteiro em três linhas: pegou o FormData, converteu em objeto com Object.fromEntries, e o preventDefault impediu o navegador de recarregar tudo. Esse padrão está em praticamente todo formulário moderno da web.

+

O detalhe que vai te morder depois

Repare que o tempo veio como string "45", não como número 45. O FormData SEMPRE retorna strings, não importa o tipo do input.
Se você tentar somar tempos depois para mostrar "total de horas estudadas", "45" + "30" vira "4530" em vez de 75. Esse erro é tão comum que tem nome em inglês: type coercion bug. Vamos resolver no próximo passo.

4. Renderizando em tela

Console é pra desenvolvedor. Aluno quer ver no caderno. Atualize o app.js:

javascript 
const form = document.getElementById('form-sessao');
const lista = document.getElementById('lista-sessoes');

form.addEventListener('submit', function(evento) {
  evento.preventDefault();

  const dados = new FormData(form);
  const sessao = Object.fromEntries(dados);
  sessao.tempo = Number(sessao.tempo);

  renderizarSessao(sessao);
  form.reset();
});

function renderizarSessao(sessao) {
  const card = document.createElement('article');
  card.className = 'sessao-card';
  card.innerHTML = `
    <h3>${sessao.materia}</h3>
    <span class="tempo">${sessao.tempo} min</span>
    <p class="anotacao">${sessao.anotacao || '<em>(sem anotação)</em>'}</p>
  `;
  lista.prepend(card);
}

Quatro mudanças importantes nesse trecho. A linha sessao.tempo = Number(sessao.tempo) converte a string em número de verdade — vamos precisar disso lá na aula de gráficos. O form.reset() limpa os campos depois de salvar. O lista.prepend(card) coloca a sessão nova no topo, em vez de empilhar no fim. E o uso de template string com crase permite injetar variáveis no HTML sem concatenar manualmente.

Recarregue, preencha "Matemática, 45 min, Função quadrática", clique em Salvar. Card aparece animado, formulário se limpa. Preenche "Física, 30 min, Cinemática", salva. Outro card no topo. Funciona.

Seção 04

Validando e depurando

✓

Você está no caminho certo se

Preencheu uma sessão de matemática com 45 minutos, clicou em Salvar sessão, a página NÃO recarregou, e um card verde apareceu animado acima do formulário com matéria, tempo e anotação visíveis. Os campos voltaram a ficar vazios. Você consegue adicionar várias sessões e elas se acumulam empilhadas, com a mais nova no topo.

!

A página recarrega e some tudo

Sintoma: você clica em Salvar e a página pisca, perdendo tudo.
Solução: esqueceu o preventDefault, ou ele está depois de um erro de sintaxe que interrompeu a execução. Abra o console antes de clicar e veja se aparece erro vermelho — conserte o que estiver acima do preventDefault. Esse erro vai acontecer com você pelo menos uma vez por semana nos próximos seis meses. Faz parte.

!

Card não aparece, mas o console mostra o objeto certo

Sintoma: o objeto sai no console, mas nenhum card surge na tela.
Solução: você está fazendo console.log mas esqueceu de chamar renderizarSessao, ou definiu a função em lugar errado. Confirme que a chamada está exatamente renderizarSessao(sessao) e que o id da section é exatamente lista-sessoes, sem espaço, sem typo.

!

Aparece a palavra Object entre colchetes

Sintoma: o card mostra [object Object] em vez dos dados.
Solução: você passou o objeto inteiro pro innerHTML em vez de acessar as propriedades. Garanta que está usando sessao.materia, sessao.tempo e sessao.anotacao no template — não a variável sessao crua.

Para quem terminou e quer mais: adicione um campo Data com input type="date", preenchido automaticamente com o dia de hoje via JavaScript no carregamento da página. Mostre a data formatada no card como 23 de maio, não 2026-05-23.

✦

Onde você acabou de usar tecnologia profissional

FormData não é didática — é o que roda o Facebook, o iFood, o Mercado Livre. Toda vez que você preenche endereço de entrega e o site não pisca, é exatamente esse padrão rodando por baixo. A diferença entre o seu código de hoje e o do iFood são camadas, não conceitos.
Em entrevistas de estágio front-end, explicar a diferença entre o submit padrão e o controlado por JavaScript é pergunta de primeira eliminatória. Você acabou de saber responder isso na prática, não na decoreba.

Questão de fixação

Antes de responder, considere o código abaixo (atenção à ORDEM das linhas):

javascript 
const form = document.getElementById('form-sessao');

form.addEventListener('submit', function(e) {
  const dados = new FormData(form);
  console.log(Object.fromEntries(dados));
  e.preventDefault();
});

O usuário preenche e clica em Salvar. O que acontece, na ordem real?

  1. O objeto aparece no console, a página fica intacta — código funciona perfeitamente.
  2. O objeto aparece no console por uma fração de segundo, depois a página recarrega. (correta)
  3. Um erro é lançado porque preventDefault deve ser obrigatoriamente a primeira linha.
  4. Nada aparece no console porque o navegador recarrega antes do console.log executar.
i

Resposta comentada

A chave é a ordem das linhas. O console.log é executado SIM (não há erro de sintaxe, e o JS roda linha por linha), então o objeto aparece no console. Mas como o preventDefault foi chamado APÓS o console.log, o navegador continua com o ciclo padrão logo na sequência: empacota a requisição e recarrega a página. Resultado prático: você vê o objeto piscar e some tudo.
Por isso a regra profissional é colocar o preventDefault como primeira linha executável do listener. A alternativa C está errada porque ele pode tecnicamente ser chamado em qualquer ponto antes do final do callback — só não convém. A D está errada porque o JS é síncrono dentro do listener: o console.log roda ANTES de o navegador continuar o ciclo padrão.

Fechamento

O que levamos desta aula

  • FormData captura todos os campos do formulário que tenham atributo name, num objeto iterável.
  • Object.fromEntries converte o FormData em objeto JavaScript manipulável — pronto pra ser salvo, exibido ou enviado.
  • preventDefault no submit interrompe o ciclo padrão do navegador e devolve o controle pro JavaScript.
  • Number converte string em número, porque o FormData sempre retorna strings, mesmo de campos numéricos.
  • Na próxima aula vamos resolver a persistência: feche o navegador agora e tudo desaparece. Vamos usar a memória secreta que todo navegador tem e quase ninguém conhece direito.
